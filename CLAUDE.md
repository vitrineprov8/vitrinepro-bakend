# vitrinepro-bakend — Guia para Claude (documento vivo)

Backend NestJS 11 que serve o frontend `../vitrinepro-frontend-v2`. **É a fonte de verdade dos contratos de API.**

> ⚠️ **REGRA OBRIGATÓRIA — LER E ATUALIZAR SEMPRE**
> 1. **Antes** de qualquer tarefa neste repo, ler este arquivo inteiro.
> 2. **Depois** de adicionar/alterar um endpoint, entidade, migração, módulo ou regra de negócio, **atualizar este arquivo no mesmo commit** (mapa de módulos §Endpoints e/ou §Regras). Um endpoint novo que não está aqui é um bug de documentação.
> 3. Gaps e regras de negócio de produto vivem em `../vitrinepro-frontend-v2/PLANO_DESENVOLVIMENTO.md` (§BACKEND, códigos B1..B16 e RN-*). Este arquivo é o "como está implementado"; aquele é o "o que falta".

## Stack
- **NestJS 11** + **TypeORM 0.3.28** + **PostgreSQL** (`pg`). Node/TS.
- Auth: `@nestjs/jwt` + Passport (`passport-jwt`, Google + LinkedIn OAuth). Hash: `bcryptjs`.
- Uploads/CV/avatar: AWS S3 (`@aws-sdk/client-s3`) + `sharp` + `pdfkit` (PDF de processo).
- Agendamento: `@nestjs/schedule`. Slugs: `slugify`. Validação: `class-validator` + `class-transformer`.

## Convenções (seguir à risca — copiar de módulos existentes)
- **Rotas sem prefixo global** (não há `setGlobalPrefix`). Controllers usam `@Controller()` vazio e declaram o path completo no método: `@Get('vagas/:id/applications')`. Rotas públicas são livres; protegidas usam `@UseGuards(JwtAuthGuard)`.
- **`req.user` é a entidade `User` completa** (retornada por `JwtStrategy.validate` → `authService.validateUser`). Tipar como `{ user: { id: string; role: UserRole } }` no controller e ler `req.user.id` / `req.user.role`. Ownership/autorização é aplicada **no service**, não no controller.
- **DTOs** com `class-validator` (`@IsString`, `@IsEmail`, `@IsUUID`, `@IsOptional`, `@MaxLength`…). `ValidationPipe` global é `{ whitelist: true, transform: true }` — campos não declarados no DTO são removidos silenciosamente.
- **Entities**: `@PrimaryGeneratedColumn('uuid')`, `@CreateDateColumn`/`@UpdateDateColumn`, FKs via `@ManyToOne` + coluna `xxxId` explícita (`@Column({ type: 'uuid' })`) + `@JoinColumn`. `jsonb` para estruturas flexíveis. Enums deprecados ficam documentados no arquivo (ver `ApplicationStatus`), não se usam em código novo.
- **Migrações** (`src/migrations/<timestamp>-<nome>.ts`): `synchronize: false` — **toda mudança de schema exige migração**. Padrão: `up`/`down` idempotentes com `IF NOT EXISTS` / `IF EXISTS`, `uuid_generate_v4()` para PKs, constraints e índices nomeados (`PK_`, `FK_`, `UQ_`, `IDX_`). Nome da classe = `Nome<timestamp>` e propriedade `name` igual. Registrar a entity nova no `TypeOrmModule.forFeature([...])` do seu módulo.
- **⚠️ Registro de entidades em 3 lugares (senão quebra):** toda entity nova precisa ser adicionada ao array `entities` de **`src/data-source.ts`** (usado pelo CLI `migration:run`) **E** de **`src/database/database.config.ts`** (runtime da app), além do `forFeature` do módulo. Esquecer a lista causa `TypeORMError: Entity metadata for X#relation was not found`.
- **Módulos**: cada feature = pasta `src/<feature>/` com `<feature>.module.ts` + `.controller.ts` + `.service.ts` + `<entity>.entity.ts` + `dto/`. Registrar o módulo em `src/app.module.ts` (`imports`). Se outro módulo precisar do repositório, exportá-lo no `exports` do módulo dono.
- **Erros**: usar exceções Nest (`NotFoundException`, `ForbiddenException`, `ConflictException`, `BadRequestException`) com mensagem em **PT-BR** (a UI mostra a mensagem). Códigos de app (ex.: `PLAN_LIMIT_REACHED`) vão no corpo quando o front precisa reagir.
- CORS liberado para `localhost:3000/4321` e domínios `v8pro.com.br` (`src/main.ts`).

## Regras de negócio implementadas (RN)
- Não se pode publicar vaga via PATCH — só `POST /vagas/:id/publish` (consome slot do plano).
- Conta empresa (`isCompany`) não pode se candidatar como candidato.
- Candidatura é única por `(vagaId, userId)`.

### B3 — Submissão de candidatos por hunter
- `hunter_candidates`: pool privado por hunter (candidato "fantasma" sem conta). Único por `(hunterId, email)`.
- Consentimento LGPD por token de e-mail (envio real via B14; stub se sem chave): status `PENDING/GRANTED/DECLINED`. Submissão exige `GRANTED`.
- `POST /vagas/:id/submissions`: exige `vaga.allowHunters`; **limite N por hunter por vaga** (RN-NOVA-01) e **trava de duplicidade `vaga.exclusivityDays`** por candidato+vaga (RN-NOVA-02, ver B4 abaixo). Cria `VagaApplication` com `source='HUNTER'`, `submittedByHunterId`, `hunterCandidateId` (e `userId` null para fantasma).

### B4 — Marketplace/fee de hunters
- **Fee obrigatório para publicar vaga aberta a hunters**: `VagasService.publish()` bloqueia (`BadRequestException`) se `allowHunters=true` e `feePercent`/`feeAmount` ambos nulos.
- **Termos de intermediação**: `POST /vagas/:id/hunter-interest` exige `termsAccepted:true` (`ExpressInterestDto`); grava `termsAcceptedAt`.
- **RN-NOVA-01/02** (limite de submissões e trava de duplicidade) continuam como no B3, mas o prazo da trava agora vem de `vaga.exclusivityDays` (configurável, default 90) em vez de constante fixa.
- **Cap de hunters aceitos**: `maxHunters` por vaga (default 5); `HunterInterestsService.updateStatus()` rejeita (`ConflictException`) aceitar além do cap.
- **RN-NOVA-03 (mascaramento de contato)**: candidaturas `source='HUNTER'` têm `snapshotEmail`/`snapshotPhone` ocultos em `GET /vagas/:id/applications` até a candidatura atingir a etapa (`PipelineTemplate` `order`) configurada em `User.hunterContactRevealStageOrder` (default 2, ajustável via `PATCH /profile`).

### B5 — Perfil público de hunter + diretório
- `User` ganha `hunterSpecialties` (`simple-array`), `hunterYearsExperience` (int nullable) — editáveis via `PATCH /profile` (self-service, T-H11).
- `GET /hunters` (diretório, T07): só usuários `isVisible && !isCompany && personas LIKE '%HUNTER%'`; filtros `specialty`/`city` (ILIKE) e `verifiedOnly` (default **true** — só `verificationStatus=APPROVED`); paginado; cada card inclui `HunterMetrics` (ver abaixo).
- `GET /hunters/:username` (perfil, T08): mesmos critérios de visibilidade do diretório; 404 se não encontrado ou não é hunter.
- `HunterMetrics` (`totalIndicacoes`/`taxaAproveitamento`/`tempoMedioAteAbordagemDias`/`avaliacaoMedia`) é computada on-the-fly de `VagaApplication` por `submittedByHunterId` — são **proxies** até B9 (placements reais) e B10 (avaliações reais) existirem; `order` do diretório com `'placements'`/`'rating'` cai para `'recent'` pelo mesmo motivo (mesmo padrão já usado em `vagas/radar` para B9/B10).

### B8 — Verificação de hunter
- `HunterVerificationStatus` (`NONE|PENDING|APPROVED|REJECTED`) + `User.verificationStatus/verificationDocs(jsonb)/verificationLinkedinUrl/verificationRequestedAt/verificationDecidedAt/verificationRejectionReason`.
- Fluxo hunter: `POST /profile/me/verification/documents` (upload N docs, sem mudar status) → `POST /profile/me/verification/submit` (`PENDING`, limpa decisão anterior) — só reenviável em `NONE`/`REJECTED` (`APPROVED` bloqueia re-envio).
- Fluxo admin: fila `GET /admin/hunters/verifications` → `approve`/`reject` (`RejectVerificationDto{reason}` obrigatório) — ambos exigem status `PENDING` (400 caso contrário) e disparam e-mail (`sendVerificationApproved`/`sendVerificationRejected`, este último com o motivo).
- **Gate do marketplace**: `hunter-interests.express()` e `hunter-candidates.submitToVaga()` bloqueiam (`ForbiddenException{code:'HUNTER_NOT_VERIFIED'}`) hunters com `verificationStatus !== APPROVED` — é o único efeito colateral de negócio da verificação hoje (o selo "Verificado" em si é só exibição no B5).

### B13 — Notificações (sino, parcial — sessões/export ficam pro B26)
- Novo módulo `notifications` (`@Global()`, mesmo padrão de `AdminAuditLogModule`/`MailModule` — qualquer service injeta `NotificationsService` sem reimportar o módulo). Migração `1750000000000-b13-notifications`.
- `Notification` (`notifications`): `userId`, `type` (enum `NotificationType`, 12 valores), `title`/`message`/`link`(nullable)/`metadata`(jsonb, nullable), `readAt`(nullable), `createdAt`. `NotificationPreference` (`notification_preferences`, `@Unique(['userId','type'])`): `inAppEnabled`/`emailEnabled` (bool, default `true`) — default por tipo em `NOTIFICATION_DEFAULT_CHANNELS` (só `STAGE_CHANGED` nasce `email:false`; os outros 11 nascem `inApp:true, email:true`).
- `NotificationsService.create(input)` é **fire-and-forget** (chamado sempre com `void`) e **nunca lança** — try/catch interno loga e retorna `null` em qualquer erro (mesmo padrão do `AdminAuditLogService.record`). Antes de gravar, olha `NotificationPreference` (fallback pro default se não houver override) e **não grava nada** se `inAppEnabled=false` — é aqui que a supressão por preferência acontece, não no controller.
- `createForEmailIfExists(email, rest)` — usado quando o alvo pode não ter `userId` ainda (convite de time por e-mail): resolve o `User` por e-mail (`select:['id']`) e só chama `create()` se existir; senão vira no-op silencioso.
- Endpoints (`JwtAuthGuard`, `@Controller('notifications')`): `GET /` (paginado, `?unreadOnly=true`, `?page`/`?limit`, retorna `{data, total, unreadCount, page, lastPage}`), `PATCH /:id/read` (idempotente, 404 se não for do próprio usuário), `POST /read-all` (`{updated: N}`), `GET /preferences` (sempre retorna os 12 tipos, merge de overrides sobre os defaults), `PATCH /preferences` (`{preferences: [{type, inAppEnabled?, emailEnabled?}]}` — upsert por tipo).
- **Conectado em 6 serviços já existentes**, sempre com `void this.notificationsService.create({...})` logo depois da ação já persistida (e logo depois do `mailService.sendXxx` já existente, quando há um):
  - `hunters.service.ts`: `adminApprove`/`adminReject` → `VERIFICATION_DECIDED`.

Corrigido com `@UseGuards(OptionalJwtAuthGuard)` (autentica se houver token, mas não exige) + `PortfolioService.findAll(dto, requesterId?)` ganhou `isOwnerQuery = !!userId && !!requesterId && userId === requesterId` — só permite ver status diferente de `PUBLISHED` quando o requisitante autenticado é o próprio dono do `userId` pedido; senão força `status=PUBLISHED` (ou 403 se pediu explicitamente outro status sem ser dono). Dois endpoints novos, **em paralelo** ao fluxo público por token de e-mail já existente (B3/B14, não substituído):

  - `POST /me/consent-requests/:id/decide` (`JwtAuthGuard`, `DecideConsentDto{decision}`) — `decideConsentAuthenticated(id, userId, decision)`: valida `candidate.linkedUserId === userId` (404 se não bate — mesmo padrão de "não vazar existência" usado em outros pontos), exige `consentStatus === PENDING` (400 se já decidido), grava `GRANTED`/`DECLINED` + `consentDecidedAt`, invalida `consentToken`.

Comportamento é intencional (consentimento já concedido não devia poder ser "re-perguntado" silenciosamente), só documentando pra não re-derivar na próxima vez que precisar gerar um cenário de teste.
- **Validado E2E completo (2026-07-12/13)** contra o backend real: criei um `HunterCandidate` novo (via conta hunter de sempre) apontando pro e-mail de uma conta candidato descartável recém-registrada — `linkedUserId` preenchido automaticamente no create (confirmado no response), `POST /request-consent` gerou um consentToken real, `GET /me/consent-requests` (autenticado como a conta candidato) retornou o `PENDING` certo, `POST /decide {decision:'GRANTED'}` → 201 → `GET /me/consent-requests` confirmou `GRANTED`+`decidedAt` preenchido.
